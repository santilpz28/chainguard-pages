The Graph Protocol

Independent security audit of a $1B+ TVL decentralized indexing protocol. Two-axis assessment covering source code (Horizon) and mainnet bytecode (21 production contracts).

Scope

The engagement covered two complementary views of the protocol:

  • Audit A — Source code (Horizon): the new module under active development. 505 files, 87.5k LOC, 99.36% test coverage from the upstream team.
  • Audit B — Mainnet bytecode: 21 production contracts already deployed on Ethereum mainnet, obtained via the Immunefi Instascope export. 27.5k LOC of contract code, solc 0.5.3 to 0.8.27.

Each axis was analyzed independently. Findings were then deduplicated and cross-referenced. A separate on-chain investigation (using the Etherscan API) verified one operational-security flag in the deployment configuration.

Methodology

The audit followed the Pashov Audit Group two-phase methodology, augmented with an in-house 12-agent walkthrough skill:

  1. Phase 1 — Automated & structural: Foundry build, Slither static analysis (1,606 raw findings across both axes), x-ray threat modeling, Etherscan on-chain history checks, Foundry coverage.
  2. Phase 2 — Manual review: 12 parallel reasoning lenses (reentrancy, access control, math, gas, etc.), line-by-line reading of every high-LOC core contract, cross-reference between tool output and invariants, non-Solidity deep audit (CI/CD, deployment scripts, hardhat keys, dependencies).

Findings

Of 1,606 raw Slither findings, only 10 issues survived the FP analysis — a 99.4% reduction rate, consistent with the protocol's maturity.

Operational Security Investigation

One of the medium findings during initial analysis was a public BIP-39 mnemonic in the legacy deployment configuration, assigned as the signer for Arbitrum One (mainnet) network. Without verification, this would have been flagged as critical.

Using the Etherscan API v2, I derived the address at BIP-44 path m/44'/60'/0'/0/0 and pulled the full transaction history. The result: 1,930 transactions on Arbitrum One, but no privileged role in The Graph protocol — the actual GraphProxyAdmin on Arbitrum One is a different address, and the activity pattern is consistent with an unrelated third-party developer who used the public Hardhat default mnemonic. The severity was downgraded from Critical to Low.

This is the value of on-chain verification over static analysis alone — Slither and grep would have flagged the public mnemonic regardless, but only by checking the actual on-chain activity can we confirm whether the threat is real.

Positive Observations

What stands out about The Graph's security posture:

  • High test coverage — 99.36% line coverage on the Horizon package, with explicit regression tests for known attack vectors (returndata bombing, replay attacks, unauthorized service attacks).
  • Active auditor engagement — Trust Security audit tags appear throughout the commit history.
  • Strong CI/CD hygiene — OIDC trusted publishing, SLSA provenance, audit-label gate on PRs to main.
  • Multisig governance — All admin operations route through a GnosisSafe.
  • Proactive secrets migration — Horizon and subgraph-service were migrated to hardhat-secure-accounts in late 2024; the legacy v5 package remains the only gap.

Recommendation Summary

The total fix effort across all 10 findings is ~4.5 hours of development work. After applying these fixes, the protocol will have no remaining defense-in-depth gaps identified by either audit. The protocol can continue operating without changes in the interim.

Methodology Reuse

This two-axis pattern (source + mainnet) is highly portable. It works particularly well for protocols that:

  • Have an Immunefi or similar bug bounty program (mainnet bytecode is downloadable)
  • Have an open-source repository (source code is auditable)
  • Have been audited previously (findings dedup against known issues)

Total time for a similar engagement: 2-4 hours for an experienced auditor, scaling with codebase size and prior audit count.
